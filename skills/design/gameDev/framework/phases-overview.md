# 游戏设计六阶段概览

> **版本**: 1.0.0  
> **最后更新**: 2026-01-14

本文档详细说明游戏设计的6个阶段，每个阶段的目标、输入输出和核心任务。

---

## Phase 1: 概念设计 (Concept Design)

### 核心目标

**定义游戏是什么、给谁玩、为什么好玩**

在这个阶段，我们从模糊的创意出发，回答三个最核心的问题：

1. **游戏是什么？** (What) - 游戏的身份、类型、核心体验
2. **给谁玩？** (Who) - 目标玩家是谁，他们的需求是什么
3. **为什么好玩？** (Why) - 核心循环、情感设计、独特价值

### 输入

- 初始游戏创意（可以是一句话、一个想法、或参考游戏）
- 目标平台/限制条件（可选）

### 输出

- **@concept.md** - 游戏概念文档
- 100个维度的设计决策文档

### 12个设计维度（100个任务）

| 维度 | 任务数 | 核心问题 |
|------|--------|----------|
| 1. 游戏身份定义 | 10 | 这个游戏是什么？如何用一句话描述？ |
| 2. 玩家定义 | 12 | 谁会玩这个游戏？他们想要什么？ |
| 3. 核心幻想 | 8 | 玩家想成为谁？想做什么？ |
| 4. 核心循环 | 10 | 玩家重复进行的基本行为链是什么？ |
| 5. Moment-to-Moment | 10 | 每一秒的游戏体验是什么？ |
| 6. 情感设计 | 10 | 游戏传递什么情感？如何传递？ |
| 7. 挑战与难度 | 8 | 挑战来自哪里？难度如何设计？ |
| 8. 进度与成长 | 8 | 玩家如何成长？进度如何体现？ |
| 9. Session 设计 | 6 | 一次游戏多长时间？如何设计？ |
| 10. 主题与世界观 | 10 | 游戏的世界是什么样的？ |
| 11. 参考与对标 | 4 | 参考哪些游戏？如何差异化？ |
| 12. 约束与范围 | 4 | 平台限制？MVP范围？风险？ |

### 关键里程碑

- ✅ 电梯演讲（30秒说清游戏）
- ✅ 核心循环定义
- ✅ 目标玩家画像
- ✅ 3个设计支柱
- ✅ MVP范围界定

### 验收标准

- [ ] 任何人读完概念文档，能理解这个游戏是什么
- [ ] 核心循环可以用简单图表表达
- [ ] 明确定义了"不做什么"（反面清单）
- [ ] 识别了最大的3个风险
- [ ] 有具体的成功指标

## Phase 2: 系统设计 (System Design)

### 核心目标

**拆解游戏系统、定义系统交互**

将抽象的概念拆解为具体的游戏系统，并定义系统之间的交互关系。

### 输入

- Phase 1 的概念文档
- 核心循环定义
- 玩家行为清单

### 输出

- **@systems-breakdown.md** - 系统拆解文档
- 系统交互图
- 每个系统的职责定义

### 核心任务（待细化）

1. **系统识别** - 识别游戏包含哪些系统
2. **系统分类** - 核心系统 vs 辅助系统
3. **系统职责** - 每个系统负责什么
4. **系统交互** - 系统之间如何通信
5. **系统优先级** - 哪些系统必须先做
6. **系统边界** - 每个系统的范围界定

### 常见系统类型

| 系统类型 | 示例 | 典型职责 |
|---------|------|---------|
| 输入系统 | 点击、拖拽、按键 | 接收玩家操作 |
| 战斗系统 | 攻击、防御、技能 | 处理战斗逻辑 |
| 经济系统 | 货币、商店、交易 | 资源生产与消费 |
| 进度系统 | 等级、解锁、成就 | 玩家成长 |
| 社交系统 | 好友、排行榜、聊天 | 玩家互动 |
| UI系统 | 界面、反馈、提示 | 信息展示 |

### 关键里程碑

- ✅ 系统列表（10-20个）
- ✅ 系统交互图
- ✅ MVP系统范围
- ✅ 系统实现优先级

## Phase 3: 机制设计 (Mechanics Design)

### 核心目标

**定义每个系统的详细规则和数值**

为每个系统设计具体的游戏机制、规则和初始数值。

### 输入

- Phase 2 的系统拆解
- 每个系统的职责定义

### 输出

- **@mechanics/[系统名].md** - 每个系统的机制文档
- **@economy.md** - 经济数值设计
- 规则文档和数值表

### 核心任务（待细化）

1. **规则定义** - 每个系统的运行规则
2. **参数设计** - 关键参数的初始值
3. **边界条件** - 极端情况的处理
4. **反馈设计** - 玩家如何感知机制
5. **平衡设计** - 数值的初步平衡

### 关键里程碑

- ✅ 核心机制规则文档
- ✅ 经济平衡表
- ✅ 进度曲线设计
- ✅ 战斗公式定义

---

## Phase 4: 技术规划 (Technical Planning)

### 核心目标

**技术选型、架构设计、实施计划**

确定技术方案，设计代码架构，制定可执行的实施计划。

### 输入

- Phase 1-3 的设计文档
- 技术约束（平台、团队技能）

### 输出

- **@tech-stack.md** - 技术选型文档
- **@architecture.md** - 代码架构设计
- **@implementation-plan.md** - 实施计划

### 核心任务（待细化）

1. **技术选型** - 引擎、框架、工具链
2. **架构设计** - 模块划分、数据流
3. **技术验证** - 关键技术的可行性
4. **任务拆分** - 开发任务的WBS
5. **依赖排序** - 任务的依赖关系
6. **风险识别** - 技术风险和缓解方案

### 关键里程碑

- ✅ 技术栈确定
- ✅ 架构图完成
- ✅ 开发任务清单
- ✅ 技术原型验证

---

## Phase 5: 内容规划 (Content Planning)

### 核心目标

**关卡、叙事、资产规格**

规划游戏的具体内容，定义资产需求和制作规格。

### 输入

- Phase 1-4 的设计和技术文档
- 内容目标（关卡数、角色数等）

### 输出

- **@content-plan.md** - 内容规划文档
- **@asset-spec.md** - 资产规格说明
- **@narrative.md** - 叙事设计

### 核心任务（待细化）

1. **关卡设计** - 关卡结构、难度曲线
2. **叙事设计** - 故事线、角色、对话
3. **资产清单** - 需要的美术、音频资产
4. **资产规格** - 每种资产的制作要求
5. **内容优先级** - MVP内容范围

### 关键里程碑

- ✅ 关卡地图
- ✅ 叙事大纲
- ✅ 资产清单
- ✅ 资产制作规范

---

## Phase 6: 验证迭代 (Validation & Iteration)

### 核心目标

**测试设计、平衡验证、迭代规划**

设计测试方案，验证设计假设，基于反馈迭代改进。

### 输入

- Phase 1-5 的所有设计文档
- 游戏原型或实现

### 输出

- **@playtest-plan.md** - 测试计划
- **@feedback-analysis.md** - 反馈分析
- **@iteration-plan.md** - 迭代计划

### 核心任务（待细化）

1. **测试用例设计** - 要测试什么
2. **测试场景设计** - 如何测试
3. **反馈收集** - 收集什么反馈
4. **数据分析** - 分析游戏数据
5. **假设验证** - 验证设计假设
6. **迭代规划** - 基于反馈的改进

### 关键里程碑

- ✅ 测试用例文档
- ✅ 完成内部测试
- ✅ 反馈报告
- ✅ 下一版本计划

## 阶段间的流转

### 瀑布式流转（适合新项目）

```
Phase 1 → Phase 2 → Phase 3 → Phase 4 → Phase 5 → Phase 6
                                                      ↓
                                          (循环回 Phase 1 或 3)
```

### 迭代式流转（适合原型验证）

```
Phase 1 → Phase 2 → Phase 4 (快速原型)
   ↑                           ↓
   └─── Phase 6 (测试) ←────────┘
         ↓
      Phase 3 (完善机制)
```

### 并行式流转（适合大型项目）

```
Phase 1 → Phase 2 → Phase 3
                 ↘         ↘
                  Phase 4   Phase 5 (并行)
                 ↗         ↗
                    Phase 6
```
