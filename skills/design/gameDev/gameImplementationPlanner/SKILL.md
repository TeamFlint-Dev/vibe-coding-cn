# gameImplementationPlanner Skill

将游戏设计文档转化为具体的代码架构和可执行的开发任务，制定清晰的实施路线图。

## When to Use This Skill

触发此 Skill 当：
- 设计和技术选型完成，准备开发
- 需要制定开发计划和任务拆解
- 需要设计代码架构
- orchestrator 调度到实施规划阶段

## Not For / Boundaries

此 Skill 不负责：
- 游戏设计内容（需要设计文档作为输入）
- 技术选型（需要 @tech-stack.md 作为输入）
- 实际代码编写（输出是计划，不是代码）
- 项目管理执行（输出是计划，执行是下一步）

必需输入：
- `@concept.md` - 游戏概念
- `@systems-breakdown.md` - 系统设计
- `@mechanics/*.md` - 机制设计（至少核心系统）
- `@tech-stack.md` - 技术选型

可选输入：
- `@economy.md` - 经济设计

## Quick Reference

### 实施规划流程

```
1. 汇总所有设计输入
2. 设计代码架构
3. 拆解开发任务
4. 确定任务依赖和优先级
5. 制定里程碑计划
6. 输出 @architecture.md + @implementation-plan.md
```

### 架构设计原则

| 原则 | 描述 |
|------|------|
| **单一职责** | 每个模块只做一件事 |
| **依赖倒置** | 依赖抽象而非具体实现 |
| **接口隔离** | 模块间通过清晰接口通信 |
| **开闭原则** | 对扩展开放，对修改关闭 |

### 任务拆解原则

| 原则 | 描述 |
|------|------|
| **SMART** | 具体、可测量、可实现、相关、有时限 |
| **垂直切片** | 每个任务产出可测试的功能 |
| **独立性** | 减少任务间依赖 |
| **适度粒度** | 1-3 天可完成 |

### 任务优先级

| 优先级 | 定义 | 示例 |
|--------|------|------|
| **P0** | 核心循环必需 | 核心玩法、基础 UI |
| **P1** | MVP 必需 | 存档、基础进度 |
| **P2** | 完整体验需要 | 音效、成就 |
| **P3** | 锦上添花 | 社交、扩展内容 |

### Example 2: 任务拆解

**输入**: @architecture.md 的收集系统模块

**任务拆解**:
```
T010: 收集系统 - 数据结构
- 定义可收集物品数据模型
- 预估: 2h
- 依赖: 无
- 验收: 数据结构可存储物品属性

T011: 收集系统 - 核心逻辑
- 实现收集判定和物品生成
- 预估: 4h
- 依赖: T010
- 验收: 可触发收集并获得物品

T012: 收集系统 - 动画反馈
- 实现收集动画和音效
- 预估: 3h
- 依赖: T011
- 验收: 收集有视觉反馈
```

**输出**: @implementation-plan.md 相关章节

### Example 3: 里程碑规划

**输入**: 所有系统的任务列表

**里程碑规划**:
```
M0 原型（1周）:
- 基础收集交互
- 单一展示槽位
- 简单收益计算
- 临时 UI

M1 MVP（2周）:
- 完整收集系统
- 多槽位展示
- 背包系统
- 基础进度
- 存档功能

M2 Alpha（2周）:
- 商店系统
- 完整进度树
- 音效音乐
- UI 美化
```

**输出**: @implementation-plan.md 里程碑章节
