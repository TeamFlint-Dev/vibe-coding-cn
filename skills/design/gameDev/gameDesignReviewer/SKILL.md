# gameDesignReviewer Skill

设计审阅与迭代专家。在 memory-bank 文档生成后，主动解释设计意图，征求用户反馈，并支持持续迭代修改，直到设计完全符合用户设想。

## When to Use This Skill

触发此 Skill 当：

- 用户说"审阅设计"、"检查一下"、"看看对不对"
- 用户说"这不是我想要的"、"不太对"、"有问题"
- 用户说"解释一下"、"为什么这样设计"
- 用户说"继续迭代"、"再改改"、"修改设计"
- 任何设计 Skill 输出后，用户表示疑惑或不满
- orchestrator 调度设计迭代阶段

## Not For / Boundaries

此 Skill 不负责：

- 生成新的设计文档（交给对应的 designer Skill）
- 审计 SKILL.md 文件质量（交给 gameSkillAuditor）
- 代码实现（交给 code-generator）

必需输入：

- 已存在的 memory-bank 文件（如 @concept.md、@economy.md 等）
- 或用户明确指定要审阅的设计内容

## Quick Reference

### 审阅迭代流程

```text
┌──────────────────────────────────────────────────────────────┐
│                    设计审阅迭代循环                           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  1. 读取文档                                                  │
│     └── 加载目标 memory-bank 文件                             │
│                                                              │
│  2. 结构化解释                                                │
│     ├── 分段展示设计内容                                      │
│     ├── 解释每个设计决策的意图                                │
│     └── 标注关键假设和依赖                                    │
│                                                              │
│  3. 主动征求意见                                              │
│     ├── 针对每个关键点提问                                    │
│     ├── 提供备选方案供选择                                    │
│     └── 等待用户反馈                                          │
│                                                              │
│  4. 记录反馈                                                  │
│     ├── 整理用户的修改意见                                    │
│     └── 确认理解无误                                          │
│                                                              │
│  5. 应用修改                                                  │
│     ├── 直接修改 memory-bank 文件                             │
│     └── 展示修改前后对比                                      │
│                                                              │
│  6. 确认或继续                                                │
│     ├── 用户满意 → 结束，更新 @progress.md                    │
│     └── 继续迭代 → 返回步骤 2                                 │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 审阅模板（每个设计段落）

```markdown
### 📖 [段落标题]

**当前设计**：
[引用文档中的具体内容]

**设计意图**：
[解释为什么这样设计，基于什么假设]

**关键问题**：
- 这符合你的设想吗？
- 如果不符合，你期望的是什么？

**备选方案**：
- A) [方案A描述]
- B) [方案B描述]
- C) 你有其他想法吗？
```

### 主动征求意见的问题库

| 设计领域 | 核心问题 |
|----------|----------|
| 核心循环 | "这个循环能让你玩得开心吗？哪个环节最吸引你？" |
| 目标玩家 | "这是你想象中的玩家吗？有没有遗漏的群体？" |
| 独特卖点 | "这个卖点够独特吗？和你心中的差异点一致吗？" |
| 系统设计 | "这些系统的优先级对吗？有没有多余或缺失的？" |
| 数值设计 | "这个数值感觉合理吗？太快还是太慢？" |
| 技术选型 | "这个技术方案你能接受吗？有没有顾虑？" |

### 迭代状态追踪

```markdown
## 迭代记录

| 轮次 | 审阅范围 | 修改点 | 用户确认 |
|------|----------|--------|----------|
| #1 | 核心循环 | 修改了奖励节奏 | ✅ |
| #2 | 目标玩家 | 添加了硬核群体 | ✅ |
| #3 | 数值设计 | 待用户反馈 | 🔄 |
```

### 反馈分类处理

| 反馈类型 | 识别特征 | 处理方式 |
|----------|----------|----------|
| 完全否定 | "不是这个"、"完全不对" | 追问期望，重新理解需求 |
| 部分修改 | "这里不对"、"改一下XX" | 定位具体点，精准修改 |
| 方向调整 | "能不能换个思路" | 提供备选方案，让用户选择 |
| 细节微调 | "差不多，但是..." | 小幅调整，确认满意 |
| 确认通过 | "可以"、"就这样" | 标记完成，进入下一阶段 |

## Examples

### Example 1: 审阅概念文档

**输入**: "审阅一下 @concept.md"

**步骤**:

1. 读取 `memory-bank/@concept.md`
2. 分段展示并解释

**输出**:

```text
## 📖 设计审阅：Trophy Fishing 概念文档

我来逐段解释这个设计，请告诉我哪里符合你的想法，哪里需要修改。

---

### 1️⃣ 一句话描述

**当前设计**：
> 一个 UEFN 放置养成游戏，玩家通过钓鱼获取战利品...

**设计意图**：
强调"放置展示"作为核心差异点，区别于普通钓鱼游戏。

**我的问题**：
- 这个描述抓住了你游戏的核心吗？
- "放置展示"是你最想突出的特色吗？
- 还是你更想强调其他方面（如收集、社交、竞技）？

---

请告诉我你的想法，我会根据反馈修改文档。
```

### Example 2: 用户表示不满意

**输入**: "这不是我想要的，我想要更社交化"

**步骤**:

1. 确认理解用户意图
2. 提供具体修改方案
3. 征求确认

**输出**:

```text
明白了，你希望**社交**成为更核心的元素。

让我确认一下你的想法：

**关于社交化，你更倾向于哪种？**

A) **展示社交**：其他玩家可以参观你的基地、点赞、留言
B) **合作社交**：多人一起钓鱼、共建基地、交易鱼获
C) **竞争社交**：钓鱼排行榜、稀有鱼争夺、赛季竞赛
D) 你有其他想法？

请选择或描述，我会更新设计文档。
```

### Example 3: 持续迭代直到满意

**输入**: "选 A 和 C 结合"

**步骤**:

1. 根据反馈修改 @concept.md
2. 展示修改对比
3. 询问是否满意

**输出**:

```text
好的，我已经更新了设计：

## 修改对比

**原核心循环**：
钓鱼 → 战利品 → 展示 → 收益 → 升级

**新核心循环**：
钓鱼 → 战利品 → 展示 → 收益 + 访客点赞 → 升级
                    ↓
              排行榜竞争

**新增内容**：
- 独特卖点增加"社交展示系统"
- 新增"排行榜"系统描述
- 目标玩家增加"社交炫耀型"

---

这个版本符合你的设想吗？
- [确认] 满意，进入下一阶段
- [继续] 还有要改的地方
```

### Example 4: 最终确认

**输入**: "可以了"

**步骤**:

1. 标记审阅完成
2. 更新 @progress.md
3. 提示下一步

**输出**:

```text
✅ 概念设计审阅完成

迭代记录：
- #1: 增强社交元素
- #2: 添加排行榜系统
- #3: 确认通过

已更新 @progress.md

下一步建议：
[1] 进入系统设计阶段
[2] 再看一遍概念文档
[3] 返回调度菜单
```
