# Failure Case Miner — 能力边界

## 能做的事（绿灯区）

| 类别 | 能力 |
|---|---|
| GitHub 数据 | 读取近 N 天 Issue/PR 标题、正文、评论来提炼失败案例 |
| 仓库扫描 | 在仓库内用 `grep` 扫描 `TODO/FIXME/HACK/WORKAROUND` 与“踩坑/注意/警告”等注释 |
| 知识沉淀 | 将结构化失败案例写入指定 `skill_path/FAILURE-CASES.md` |
| 追踪 | 对需要人工确认或信息不全的案例创建 Issue（可控数量） |

## 不能做的事（红灯区）

| 类别 | 限制 |
|---|---|
| 真实性保证 | 无法保证 Issue/PR 中叙述完全正确；必须在案例里标注来源与不确定点 |
| 外部系统 | 不能直接访问你私有的外部知识库/文档系统（除非工作流显式配置 MCP/网络权限） |
| 自动修复 | 本 Work Unit 的目标是“沉淀经验”，不默认做代码修复或重构（除非明确要求） |

## 待验证（未知区）

| 类别 | 不确定能力 | 验证方法 |
|---|---|---|
| Safe Outputs 行为 | `create-pull-request` 在本仓库的权限与分支策略下是否可用 | 用一次小改动（只更新 `FAILURE-CASES.md`）触发并观察是否能创建 PR |
