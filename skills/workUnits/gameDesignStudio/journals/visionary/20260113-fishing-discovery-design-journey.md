# 设计手记 - 发现钓点的感受设计之旅

> **日期**: 2026-01-13  
> **设计者**: Visionary  
> **任务**: IDLE-F-01 - 定义"发现钓点"的感受流程  
> **历时**: Phase 0-3 完成，约 6 小时

---

## 设计初心

当我接到"设计钓点发现"的任务时，我的第一反应是：**这不应该只是一个"标记"**。

如果玩家看到的是悬浮在空中的图标，按 E 就能钓鱼——这太没意思了。钓鱼的乐趣不仅在于"钓"，还在于"找"。真正的渔夫不会有地图告诉他哪里有鱼，他们通过观察水面、倾听声音、注意鸟类来"读水"。

**我想让玩家也体验这种"发现"的乐趣。**

---

## 关键顿悟时刻

### 顿悟 1: "发现"本身就是目标感受

在 Phase 0 审查设计模式时，我意识到：**发现不是手段，而是目的**。

大多数游戏把"发现钓点"当作"钓鱼"的前置步骤——一个必须快速跳过的环节。但我认为，发现本身就应该是一个完整的感受体验。

这让我确定了三阶段设计：
- **好奇**: "那是什么？"
- **靠近**: "去看看吧"
- **确认**: "就是这里！"

每个阶段都有自己的感受目标，而非只是"走到那里"。

### 顿悟 2: 渐进式披露创造期待感

最初我考虑过"距离足够近就显示"的简单方案。但在画感受强度曲线时，我突然想到：

**如果把信息分层披露，期待感会逐渐积累。**

```
强度 %
60  │                    ╱───
    │                 ╱
40  │              ╱
    │           ╱
20  │     ╱────
    │  ╱
0   └─────────────────────
    15m  10m  8m   5m  (距离)
```

这条曲线不是线性的，而是先平缓、再加速、最后稳定。这符合人类发现事物的自然节奏：
1. 首先注意到异常（平缓上升）
2. 靠近时好奇心增强（加速上升）
3. 确认时满足感稳定（趋于稳定）

### 顿悟 3: 环境叙事比UI更有说服力

在 Phase 3 设计叙事时，我问自己：**玩家为什么相信"这里有鱼"？**

如果只是因为有个UI图标，那这是"游戏告诉我的"。但如果是因为：
- 水面有反光（视觉证据）
- 鸟类在盘旋（生物行为）
- 水流声音特别（听觉证据）
- 能看到鱼影（直接证据）

那玩家会觉得"这里真的有鱼"，而非"这是个游戏标记"。

这个顿悟让我坚定了"环境融入"的设计方向。

## 设计挑战与解决

### 挑战 1: 如何量化"感受强度"？

**问题**: 什么是"20% 的好奇感"？如何确保玩家真的能感受到？

**挣扎**: 我纠结了很久。感受是主观的，如何用数字描述？

**解决**: 
- 使用**相对强度**而非绝对值（相对于确认阶段的 100%）
- 用**时间节奏**作为辅助（2-3 秒的好奇，3-5 秒的靠近）
- 用**可视化曲线**帮助理解
- **承认需要测试**：在审查请求中标注为"特别关注点"

**学到的**: 量化感受不是为了精确，而是为了**建立可比较的框架**。

### 挑战 2: 首次发现强化多少合适？

**问题**: +10%？+20%？+50%？没有标准答案。

**思考过程**:
1. 太低（+5%）: 玩家感觉不出差异
2. 太高（+50%）: 重复发现显得平淡无奇
3. 参考案例: 塞尔达的神庙发现、怪物猎人的素材获取

**决策**: +10-20%，具体根据频率调整
- 高频发现（每分钟）: +10%
- 低频发现（每 5-10 分钟）: +20%
- 稀有发现（每小时）: 可以更高

**学到的**: 设计参数需要**根据上下文调整**，没有万能数值。

### 挑战 3: 如何处理"玩家错过"的情况？

**问题**: 三阶段设计意味着玩家可能快速跑过而错过钓点。

**纠结**: 要不要强制让玩家注意到？

**最终决定**: **允许错过**

**理由**:
1. 钓点不稀缺，错过一个不影响游戏
2. "可以错过"反而让发现更有价值
3. 强制注意会破坏自由探索感
4. 确认音效可以作为"错过提示"（玩家经过时听到）

**学到的**: 有时候"不完美"反而更真实。不是所有东西都要确保玩家 100% 看到。

## 意外的发现

### 意外 1: Socratic Questioning 的威力

在 Phase 1 使用 Socratic Questioning 深度追问时，我原本只是遵循流程。但我发现，这些问题真的能暴露设计的盲区：

**Q: 为什么不直接显示 UI 提示？**
→ 让我思考"发现"的本质是什么

**Q: 20% 的强度会不会太弱？**
→ 让我意识到需要调整暗示的明显度

**Q: 首次和重复有何区别？**
→ 引出了差异化设计的完整方案

**学到的**: 问题比答案更重要。好的问题能引导出更深的设计。

### 意外 2: 可视化的沟通价值

我在感受定义中画了强度曲线图：

```
强度 %     首次发现
100 │                              ◆──
    │                           ╱
 80 │                        ╱
    │                     ╱
 60 │                  ╱
    │               ╱
 40 │            ╱
    │         ╱
 20 │      ╱──
    │   ╱
  0 └──────────────────────────────────
    0s  2s  4s  6s  8s  10s (时间)
```

这个简单的 ASCII 图表，比文字描述有效得多。它让抽象的"感受渐变"变得具象。

**学到的**: 设计师应该善用可视化。图表、时间线、流程图都是沟通的利器。

### 意外 3: 叙事的约束力

在 Phase 3 设计世界观时，我原本担心叙事会"束缚"设计。但实际上，叙事反而**帮助设计**：

**问题**: 为什么钓点固定在那些位置？
**叙事答案**: 因为水流交汇、岩石结构等地理因素
→ 这让钓点的布局有了合理性

**问题**: 为什么玩家能"看到"钓点？
**叙事答案**: 因为学会了像渔夫一样"读水"
→ 这让环境暗示有了意义

**学到的**: 好的叙事不是限制，而是设计的**合理化支撑**。

## 如果重新设计

### 我会保留的

1. **三阶段框架**: 这是核心，经得起推敲
2. **环境融入优先**: 沉浸感的关键
3. **差异化设计**: 首次/重复的区分很重要
4. **Socratic Questioning**: 有效的自我审查方法

### 我可能会调整的

1. **感受强度数值**: 20%、60%、100% 可能需要实际测试后调整
2. **资产需求**: 7+7+4 = 18 个资产可能偏多，看能否简化
3. **传说钓点**: 月光湖的设计可能太复杂，考虑后期加入

### 我想探索的

1. **动态钓点**: 钓点的位置/状态是否可以随时间变化？
2. **玩家学习曲线**: 如何让玩家从"不会找"到"一眼就看出"？
3. **多人协作发现**: 如果是多人游戏，发现机制如何调整？

## 核心洞察

### 洞察 1: 发现是一个动词，不是名词

"发现钓点"不是一个**状态**（找到了/没找到），而是一个**过程**（正在发现）。

设计应该关注过程的体验，而非结果的达成。

### 洞察 2: 微弱暗示比明显标记更有价值

玩家会记住"我自己发现的"，而不会记住"游戏告诉我的"。

即使微弱暗示可能被错过，也比明显标记更能创造成就感。

### 洞察 3: 量化是手段，感受是目的

20%、60%、100% 这些数字不是目的，它们只是帮助我们**建立框架**和**沟通设计**的工具。

最终的判断标准只有一个：**玩家确实感受到了吗？**

## 给未来的自己

### 记住这些

1. **先问"为什么"再问"怎么做"**: FD-012 的决策过程证明了这一点
2. **边界情况往往暴露设计盲区**: 快速经过、多钓点、重复访问等情况很重要
3. **可视化你的思考**: 曲线图、时间线、流程图都是好工具
4. **允许不完美**: 可以错过、可以简化、可以调整

### 避免这些

1. **不要纠结完美数值**: 20% 还是 25% 不是关键，测试后再调
2. **不要过度设计**: 18 个资产可能太多了
3. **不要忽视实现成本**: 有些设计很美，但实现太贵

### 继续探索

1. 这个"发现式交互"模式能否应用到宝箱、NPC、解谜？
2. 如何设计"玩家学习曲线"，从新手到专家？
3. 多人游戏中的发现机制如何调整？

## 最大收获

Phase 0 的模式审查、Phase 1 的 Socratic Questioning、Phase 3 的世界观检查——所有这些都在帮我**问问题**。

问题越深，设计越好。
