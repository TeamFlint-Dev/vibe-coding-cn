# 任务拆解设计 - 从100到122的思考旅程

> **日期**: 2026-01-13  
> **设计者**: Visionary  
> **项目**: Fishing 游戏  
> **主题**: 如何系统性地将游戏设计拆解为可执行任务

---

## 设计挑战

用户的需求看似简单：

> "为钓鱼状态机的每个状态布置设计任务，覆盖游戏所有阶段，不少于100个。完成这些任务后，游戏设计就算完成。"

但背后有几个深层问题：

1. **如何确保完整性**？怎么知道所有该设计的东西都设计了？
2. **如何确保可执行性**？100+个任务如何不变成空泛的清单？
3. **如何让进度可见**？怎么让人一眼看出设计完成了多少？
4. **如何确保质量**？如何避免为了凑数而拆分任务？

---

## 思考过程

### 第一反应：按功能拆解

最开始想到的是传统方法：
- 钓鱼流程设计
- UI设计
- 音效设计
- 美术资源设计
- 数值平衡设计
- ...

**问题**：这样拆太粗了，可能20-30个任务就拆完了，而且容易遗漏某些状态的某些维度。

### 第二思考：按状态拆解

看到状态机有7个状态，想：每个状态拆10-15个任务，就有70-105个了。

但立即发现问题：**怎么确定每个状态应该有哪些任务？**

### 顿悟：矩阵式拆解

回想到 Game Design Studio 的核心方法论——**感受驱动设计哲学**，其中有五个设计维度：

1. F - 感受定义（Feeling）
2. A - 氛围营造（Atmosphere）
3. R - 仪式设计（Ritual）
4. S - 故事包裹（Story）
5. P - 无瑕打磨（Polish）

**灵光一现**：如果为每个状态都应用这五个维度，就形成了一个矩阵！

```
        F    A    R    S    P
IDLE    │    │    │    │    │
CAST    │    │    │    │    │
WAIT    │    │    │    │    │
HOOK    │    │    │    │    │
SUCC    │    │    │    │    │
FAIL    │    │    │    │    │
MISS    │    │    │    │    │
```

这个矩阵有 **7 × 5 = 35 个单元格**，每个单元格至少1个任务，就有35+个任务了。

但更重要的是：**这个矩阵确保了完整性**——如果某个单元格是空的，立即就能看出来！

---

## 设计决策

### 决策1: 采用状态×维度的矩阵式拆解

**理由**：
- ✅ 完整性有系统保证
- ✅ 符合工作室的设计方法论
- ✅ 可追踪性强（进度可量化）
- ✅ 协作友好（可按维度分工）

### 决策2: 每个单元格的任务数不均匀

认识到不是每个单元格都应该有相同数量的任务：

- **感受定义 (F)**: 每个状态2-4个任务（定义核心感受、差异化设计等）
- **氛围营造 (A)**: 每个状态2-7个任务（视觉、听觉、交互三位一体）
- **仪式设计 (R)**: 每个状态2-4个任务（仪式流程、时间节奏等）
- **故事包裹 (S)**: 每个状态1-3个任务（叙事整合）
- **无瑕打磨 (P)**: 每个状态1-2个任务（瑕疵识别和修复）

**重点状态有更多任务**：
- SUCCESS: 20个任务（最重要的满足时刻）
- WAITING: 18个任务（悬念营造需要细节）
- HOOKED: 16个任务（惊喜爆发需要完整设计）
- IDLE: 15个任务（第一印象很重要）

### 决策3: 添加转换和全局任务

矩阵只覆盖了单个状态，但还需要：

1. **状态转换设计**（12个任务）
   - 感受衔接
   - 氛围过渡
   - 仪式缓冲
   - 叙事连续
   - 边界处理

2. **全局系统设计**（11个任务）
   - 进度与成长（图鉴、升级）
   - 教学与引导
   - UI/UX 一致性
   - 数值与平衡
   - 可扩展性

最终公式：**任务数 = 状态任务 + 转换任务 + 全局任务 = 99 + 12 + 11 = 122**

### 决策4: 优先级分配遵循金字塔原则

122个任务不是一次性完成的，需要分优先级：

- **P0 (42个，34%)**: MVP核心循环可玩
- **P1 (58个，48%)**: 完整体验
- **P2 (22个，18%)**: 细节打磨

**原则**：
- P0只包含"核心循环可玩"的任务
- P1包含"完整游戏体验"的任务
- P2包含"锦上添花"的任务

## 设计亮点

### 1. 完整性检查变得简单

传统方法很难回答"我遗漏了什么吗？"

矩阵式方法：**只要看哪个单元格是空的**。

```
✅ IDLE-F-01: 定义"发现钓点"的感受
✅ IDLE-F-02: 定义钓点"吸引力"的感受
✅ IDLE-F-03: 定义多钓点场景的选择感受
✅ IDLE-A-01: 设计钓点的视觉识别系统
❌ IDLE-A-02: (缺失) → 立即发现问题！
```

### 2. 进度追踪变得可视化

传统方法："设计大概完成了70%"（拍脑袋）

矩阵式方法：
```
总体进度: 10/122 (8%)
按状态:   IDLE 10/15 (67%), CASTING 0/12 (0%)
按维度:   F 3/23 (13%), A 4/32 (13%)
按优先级: P0 6/42 (14%), P1 4/58 (7%)
```

### 3. 协作分工变得清晰

传统方法：不知道谁负责什么

矩阵式方法：
- Visionary: 所有 F 和 S 维度任务
- 美术: A 维度的视觉任务
- 音效: A 维度的听觉任务
- 交互: R 维度任务
- 系统: P 和 G 维度任务

### 4. 产出清单一目了然

完成所有任务后，会产出：
- **52 个设计文档**（7状态×7类型+转换+全局）
- **173+ 实施资产**（动画、音效、UI、文本、配置）

这让"设计完成"有了明确的交付标准。

## 踩过的坑

### 坑1: 最初想凑整数

一开始想：能不能刚好100个？

**问题**：为了凑100，要么强行合并任务（损失完整性），要么强行拆分任务（过度细化）。

**领悟**：**让矩阵自然产生任务数，而不是反过来迁就数字**。

最终122个是自然产生的，不是强行凑的。

### 坑2: 想让每个单元格任务数相同

一开始想：每个单元格2个任务，整整齐齐。

**问题**：不同维度的复杂度不同！
- 感受定义：需要深思熟虑，2-4个任务合理
- 氛围营造：视听交互三位一体，可能需要7个任务
- 故事包裹：可能1-2个任务就够了

**领悟**：**尊重真实的复杂度，不要为了形式牺牲实质**。

### 坑3: 忘记转换和全局

一开始只拆了7个状态×5个维度 = 35单元格。

**问题**：状态间的过渡怎么办？跨状态的系统怎么办？

**领悟**：**状态机不只有状态，还有转换；游戏不只有机制，还有系统**。

## 设计原则提炼

从这次任务拆解中，提炼出几个可复用的原则：

### 原则1: 矩阵式思维

**复杂系统的拆解 = 维度1 × 维度2 × ...**

对于游戏设计：
- 维度1：系统的状态/模块
- 维度2：设计的层次（感受/氛围/仪式/叙事/打磨）

### 原则2: 完整性优先于数量

**不是"凑够多少个任务"，而是"确保没有遗漏"**

矩阵的价值不在于产生任务，而在于**暴露遗漏**。

### 原则3: 粒度由产出定义

**一个任务 = 一份完整的设计文档**

不是"设计一个按钮"（太细），也不是"设计整个UI"（太粗），而是"设计某个状态的UI展示方案"（刚好）。

### 原则4: 优先级遵循价值分布

**金字塔原则：P0 (30-40%) → P1 (40-50%) → P2 (10-20%)**

这符合价值的自然分布：
- 20%的任务创造80%的价值（P0）
- 60%的任务完善体验（P1）
- 20%的任务锦上添花（P2）

## 未来应用

这套方法可以推广到其他项目：

### 适用场景
- ✅ 有明确状态机的系统设计
- ✅ 需要多人协作的复杂项目
- ✅ 需要严格交付的商业项目

### 不适用场景
- ❌ 极简单的系统（< 20任务）
- ❌ 概念验证阶段（快速原型）
- ❌ 探索性设计（未定型）

### 改进方向
- **AI 辅助拆解**：根据状态机自动生成任务清单
- **动态优先级**：根据测试反馈调整优先级
- **模板库**：积累不同系统的拆解模板

## 收获

### 对设计的理解

**设计不是灵光一现，而是系统工程**。

通过矩阵式拆解，我意识到：
- 好的设计方法论可以产生结构
- 结构可以保证完整性
- 完整性是质量的基础

### 对方法论的理解

**感受驱动设计不只是"想清楚感受"，更是一套完整的设计体系**。

F-A-R-S-P 五个维度：
- F：定义目标
- A：传递感受
- R：强化体验
- S：包裹意义
- P：消除瑕疵

这不是装饰，而是**缺一不可的完整流程**。

### 对工具的理解

**好的工具让复杂变得可管理**。

这次创建的三份文档：
1. **任务主索引** - 清单和细节
2. **设计全景图** - 结构和关系
3. **任务追踪系统** - 管理和协作

它们形成了一个完整的系统，让122个任务变得可控。
